# V4.3 Worker Frame 与 Room Command Tick 边界

## 目标

修正 Worker 在 tick 前直接处理所有消息导致的时序问题。

本版本将 IO 消息进入 Room 的边界显式化：Worker 每轮 drain 到的消息先转换为带时间戳的 `Room::Command`，Room tick 只消费 tick start 之前已经进入 command queue 的命令。

## 核心规则

- Worker 每轮 loop 使用同一个 `frame_time` 处理本帧 drain 到的消息。
- `Room::Command::entered_at` 记录命令进入 Room command queue 的 Worker frame 时间。
- Room tick 使用自己的 `next_tick_time()` 作为 `tick_start`。
- 当前 tick 只消费 `entered_at < tick_start` 的命令。
- `entered_at >= tick_start` 的命令保留到后续 tick。
- JOIN、RECONNECT、INPUT、LEAVE 都进入 Room command queue。
- Room tick 后返回 `Room::TickResult`，Worker 根据 tick result 发送 IO 消息。

## Worker 行为变化

- `HandleJoin()` 不再直接创建 PlayerEntity，也不再立即返回 `JOIN_OK`。
- `HandleReconnect()` 不再立即返回 `RECONNECT_OK`。
- `HandlePlayerInput()` 不再直接进入旧 input queue，而是生成 `Room::Command`。
- `HandleLeave()` 不再直接删除 PlayerEntity，也不再立即修改 room 生命周期。
- Worker 仍负责 session/generation 校验、room 路由、IO 输出投递。

## Room 行为变化

- Room 新增 `Room::Command` 和 `Room::TickResult`。
- Room 在 tick 开始时切分 command queue：
  - 当前 tick 可消费命令。
  - 后续 tick 延迟命令。
- Room 是玩家实体变化、输入应用、snapshot 生成的唯一入口。
- Room 只返回 tick result，不直接访问 IO 队列。

## Pending 与 Active Session

JOIN 到达 Worker 后会先建立 pending session-room 绑定，用于保留 room 容量和后续路由。

只有 Room tick 消费 JOIN command 并产出 `JoinAccepted` 后，Worker 才会把该 session 激活为 active session。只有 active session 会收到普通 snapshot 广播。

这避免了 pending join 在正式进入 room 前收到旧 snapshot。

## 保留项

- `JOIN_OK` / `RECONNECT_OK` 携带 snapshot，后续同 tick 普通 `SNAPSHOT` 是否去重，留到网络同步输出策略统一设计。
- outbound reliable/snapshot 队列策略仍不在本版本处理。
- Worker drain 当前仍取 mailbox 当前批次，不在本版本实现 per-frame 最大消息数预算。
