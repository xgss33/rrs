# V2.2 - Multi-Worker Multi-IO Queue Topology

## Goal

V2.2 replaces the V2.1 transitional single `MessageChannels` object with explicit multi-worker/multi-IO queues.

This iteration focuses on queue ownership and routing boundaries only. It does not introduce epoll, eventfd, or the global session registry yet.

## Completed

- Replaced the old single-channel topology:

```text
inbound_to_worker
outbound_to_io
```

- Added the V2 queue topology:

```text
worker_inbound[worker_id]
worker_to_io[worker_id][io_thread_id]
```

- `MessageChannels` now owns:
  - one inbound queue per worker
  - one outbound queue per worker per IO thread

- `IOThread` now owns an explicit `IoThreadId`.

- `WorkerThread` now owns an explicit `WorkerId`.

- IO-to-worker routing now uses room ownership:

```text
WorkerId = (room_id - 1) % worker_count
```

- Worker-to-IO routing now uses the IO owner carried by inbound messages:

```text
InboundMessage.io_thread_id -> worker_to_io[worker_id][io_thread_id]
```

- Removed the server-side fake stale input injection from `RECONNECT`.

## Current Flow

### Client Input

```text
Client
  -> IOThread
  -> parse command
  -> RouteWorker(room_id)
  -> worker_inbound[worker_id]
  -> WorkerThread
  -> Room pending inputs
  -> Room tick
```

### Snapshot Output

```text
Room tick
  -> WorkerThread generates snapshot
  -> bound session carries io_thread_id
  -> worker_to_io[worker_id][io_thread_id]
  -> IOThread drains queues addressed to itself
  -> client outbound queue
  -> socket send
```

## Intentional Transitional Limits

- `Acceptor` still assigns accepted fds to the first IOThread only.
- `SessionManager` is still local to each IOThread.
- `SessionRegistry` with shared lock is not implemented yet.
- IOThread still uses polling over owned fds, not epoll.
- Acceptor-to-IO handoff still uses the existing queue, not eventfd wakeup.
- Worker-to-IO routing currently relies on `InboundMessage.io_thread_id`; later V2 should replace this with global session routing:

```text
SessionId -> IoThreadId
```

## Why This Step Matters

V2.1 had multiple IO/worker thread instances, but the message path was still logically single-channel. That means the topology existed but ownership boundaries were not yet represented in the queue layer.

V2.2 makes the queue boundaries explicit:

- each worker has a private inbound queue
- each IOThread drains only outbound queues addressed to itself
- no IOThread competes with another IOThread for the same outbound queue
- worker output no longer goes through a shared global outbound queue

This prepares the project for later V2 steps:

- global session registry
- Acceptor round-robin IO assignment
- eventfd wakeup
- epoll-based IO
