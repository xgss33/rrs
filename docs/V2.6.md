# V2.6 - epoll IO And Real RESUME

## Goal

V2.6 completes the V2 runtime direction:

- IOThread uses epoll for eventfd and client socket fds.
- The old same-connection `RECONNECT` command is removed from runtime code.
- Real new-connection resume is exposed as:

```text
RESUME <session_id>
```

## Completed

- Added per-IOThread epoll fd.
- Registered each IOThread eventfd into epoll.
- Registered accepted client fds into the owning IOThread epoll.
- IOThread now uses short-timeout `epoll_wait(..., 1ms)`:
  - Acceptor-to-IO accepted fd handoff still wakes through eventfd.
  - Worker-to-IO outbound messages do not wake IOThread directly.
  - IOThread periodically drains Worker outbound queues after each timeout or IO event.
- Replaced the old loop-level client map polling with epoll dispatch:

```text
epoll_wait
  -> eventfd readable
      -> DrainWakeEvent
      -> AcceptNewClients
      -> ProcessOutboundMessages
  -> client fd readable / hup / error
      -> ReadClientInput
      -> HandleLine
      -> CloseClient when needed
  -> timeout
      -> ProcessOutboundMessages
      -> FlushAllClientOutbound
```

- Client fd write readiness is enabled only while the per-client outbound queue still has unsent data:

```text
SendLine / snapshot queued
  -> FlushClientOutbound()
  -> send returns EAGAIN
  -> enable EPOLLOUT
  -> epoll wakes when fd is writable
  -> flush remaining bytes
  -> disable EPOLLOUT when queue becomes empty
```

- Removed runtime `RECONNECT` command handling.
- Added `RESUME <session_id>` handling.
- `SessionRegistry::Resume()` increments generation and updates:

```text
SessionId -> IoThreadId
```

- `SessionRegistry::MarkDisconnected()` now receives the IOThread owner id and only marks disconnected if the closing fd still belongs to the current owner IOThread.
- This prevents a stale old connection close from disconnecting a session that has already resumed on another IOThread.
- Worker receives a new `BindPlayer` message after resume, so generation validation continues to work.

## Current Command Set

```text
JOIN <room_id> <player_id>
INPUT <move_x> <move_y>
RESUME <session_id>
QUIT
```

## Resume Flow

```text
Client A connects
  -> JOIN room player
  -> SessionId created
  -> Client A disconnects
  -> SessionRegistry marks disconnected

Client B connects, possibly on another IOThread
  -> RESUME session_id
  -> SessionRegistry generation++
  -> SessionId -> IoThreadId updated
  -> IOThread local cache stores resumed session
  -> Worker receives BindPlayer with new generation
```

## Current IO Ownership

Each IOThread owns:

- epoll fd
- eventfd
- accepted fd queue
- assigned client fds
- fd-local session binding
- local session cache snapshot
- per-client outbound queue

## Remaining Limitations

- Worker-to-IO output no longer writes IOThread eventfd. Worker only pushes to `worker_to_io`; IOThread drains outbound queues on its short epoll timeout.
- Acceptor owns the IOThread target list for accepted-fd dispatch. It pushes the accepted fd into the selected IOThread pending queue, then writes that IOThread's eventfd directly.
- The text protocol is still temporary.
- Protobuf and binary snapshot framing are not part of V2.
- No embedded test client is added to the server project.
