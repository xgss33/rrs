# 跨局压测资源残留问题复盘

## 背景

服务器单局时长恢复为 10 分钟后，压测客户端增加了跨局运行能力：收到带有胜者的 Snapshot 后发送 LEAVE，关闭旧 TCP 连接，等待 2 秒，再用相同 PlayerId 建立新连接并重新 JOIN。随后使用 1 个 IO 线程、2 个 Worker 线程和 1024 个客户端进行了 30 分钟测试。

本次测试原本用于建立长时间性能基准，结果却暴露出旧对局资源没有释放的问题。相关原始材料：

- [服务器日志](../../logs/20260722_120722_io1_worker2.log)
- [问题发生期间的火焰图](../../perf/flamegraphs/20260722_122659_4c4g_conn1024_io1_worker2.svg)

日志和火焰图是本地分析材料，可能不会随仓库提交；文件名保留在文档中，用于从远端归档或原始测试记录中定位同一次实验。

## 问题表现

服务器配置中每个房间有 1024 个食物，房间容量为 16。1024 个客户端正常应建立 64 个房间，因此稳定运行时静态实体应约为：

```text
64 × 1024 = 65536
```

实际日志中，静态实体每经过一局便增加 65536：

| 阶段 | 静态实体 | 动态实体均值 | CPU 均值 | Tick P99 | Deadline miss |
|---|---:|---:|---:|---:|---:|
| 第一局 | 65536 | 2725 | 127% | 25.2 ms | 0.23% |
| 第二局 | 131072 | 5528 | 158% | 37.3 ms | 2.16% |
| 第三局 | 196608 | 约 8500 | 180% | 42.0 ms | 4.23% |
| 第四局开始 | 262144 | 约 10000 | 193% | 41.9 ms | 4.08% |

关键时间点如下：

```text
12:10:27  1024 connections,  65536 static entities
12:20:27     2 connections,  65536 static entities
12:20:32  1024 connections, 131072 static entities
12:30:33  1025 connections, 196608 static entities
12:40:34  1025 connections, 262144 static entities
12:41:29     1 connection,  262144 static entities
```

客户端退出后网络收发已经停止，但服务器仍保留约 9364 个动态实体并持续消耗约 92% CPU。这说明残留的不只是分配容量，而是仍在参与 Tick 的完整 Room 和玩家状态。

## 如何定位

### 1. 先检查指标之间的不变量

静态实体每次精确增加 65536，与一轮 64 个房间的食物总数完全相同。这比单纯观察 RSS 增长更有指向性：不是容器没有归还内存，也不是分配器缓存，而是上一轮所有 Room 都仍然存在。

同时，TCP 连接数在换局时先下降再恢复，说明客户端确实关闭了旧连接并建立了新连接。由此可以缩小范围：

```text
旧 TCP 已关闭
但旧 Session、Player 或 Room 没有完成清理
```

### 2. 使用火焰图确认残留资源仍在工作

火焰图采集于第二局，当时服务器已经同时保留两代房间。主要调用占比为：

- IO 线程约 41.8%；
- 两个 Worker 合计约 58.1%；
- `Room::BuildSnapshotUpdates` 合计约 30%；
- `PlayerVisibilityTracker::UpdateForObserver` 合计约 21.4%；
- `SnapshotDeltaTracker::BuildUpdate` 合计约 7.9%。

比赛结束后，Room 会停止移动、碰撞和吞噬，但仍会处理命令、计算可见性并构造 Snapshot 差异。旧房间没有连接可发送数据，却仍然承担可见性与同步计算，因此 CPU 和 Tick 尾延迟随轮次累积。

这也说明该火焰图不能作为正常 1024 连接负载的最终优化基准：它混入了整整一代已经结束但尚未清理的房间。

### 3. 沿 LEAVE 生命周期检查源码

正常清理链路应为：

```text
客户端发送 LEAVE
    → IOThread::HandleLeave
    → SessionRegistry::Remove
    → Worker 收到 kLeave
    → Room::LeavePlayer
    → WorkerRoomRegistry::RemoveSession
    → 最后一名玩家离开后删除 Room
```

Room 和 WorkerRoomRegistry 的删除条件本身是完整的。问题变成：客户端统计了 `leave_sent`，为什么服务器没有进入 `IOThread::HandleLeave`？

测试前已经修复了“比赛结束的房间仍可能接受新 JOIN”：Room 在产生 winner 的同一个 Tick 会从可加入房间列表移除。这项修改保证新一局不会进入旧对局，但不会主动删除旧 Session。由于本次 LEAVE 全部丢失，旧房间的 `session_count` 始终保持满员状态，新 JOIN 只能创建下一批房间，因此静态实体才会每局精确增加 65536。

### 4. 用单连接 Info 日志建立最小复现

修复前运行 1 个客户端 3 秒并正常退出，服务器日志只出现：

```text
client connected
join requested
join player
client closed
```

缺少：

```text
leave requested
leave player
delete empty room
```

随后指标为：

```text
connections=0 static_entities=1024 dynamic_entities=1
```

这证明问题发生在 IO 收到 TCP 字节到分发 LEAVE 之间，而不是 Worker 处理 LEAVE 之后。

### 5. 用发送字节数证明客户端确实发出了 LEAVE

单连接客户端最终记录：

```text
join_sent=1
input_sent=74
leave_sent=1
bytes_out_total=758
```

根据当前协议：

```text
JOIN  = 5 字节帧头 + 8 字节 payload = 13 字节
INPUT = 5 字节帧头 + 5 字节 payload = 10 字节
LEAVE = 5 字节帧头 + 0 字节 payload = 5 字节

13 + 74 × 10 + 5 = 758
```

发送总字节与包含 LEAVE 的理论值完全一致，因此不能把问题归因于客户端只增加了统计但没有执行 `send`。

## 根因

`IOThread::ReadClientFrames` 原本先循环调用 `recv`，直到 `EAGAIN`，然后统一解析 `read_buffer`。旧代码在以下情况直接返回失败：

- `recv` 返回 0，表示对端有序关闭；
- `recv` 返回连接重置类错误。

典型过程为：

```text
recv → 成功读取完整 LEAVE，写入 read_buffer
recv → 对端关闭或连接重置
函数立即返回失败
HandleSocketEvent 关闭 ClientConnection
read_buffer 中尚未解析的 LEAVE 被销毁
```

最初只处理了 `recv == 0` 的有序 EOF，但本地复现仍然失败。原因是客户端发送完 LEAVE 后立即关闭，同时没有继续读取服务器已经在发送的 Snapshot。Linux 在这种情况下可能向服务器报告连接重置，而不是有序 EOF。

TCP 连接随后关闭，并不否定此前已经由 `recv` 成功交付给应用层的字节。服务器必须先处理已经完整接收的帧，再结束连接生命周期。

## 考虑过的解决方法

### 方法一：客户端发送 LEAVE 后延迟关闭

客户端可以发送 LEAVE 后等待固定时间，或者继续读取一段时间再关闭。

优点：

- 修改简单；
- 可以降低 LEAVE 与连接关闭出现在同一次服务器读取中的概率。

没有采用的原因：

- 固定延迟不能提供正确性保证；
- 不同负载、网络和调度环境需要不同等待时间；
- 服务器仍然会错误丢弃在断开前已经完整收到的其他帧；
- 把真实的 TCP 边界处理缺陷转移给客户端规避。

### 方法二：增加 LEAVE_ACK 协议

客户端等待服务端确认 LEAVE 已处理后再关闭连接。

优点：

- 客户端能够明确知道服务器已经接受 LEAVE；
- 可以为更复杂的退出流程提供确认语义。

没有采用的原因：

- 需要增加协议消息类型和客户端状态；
- 当前 LEAVE 不需要业务响应，问题只在服务器错误处理已接收字节；
- 即使增加 ACK，服务器读取边界缺陷仍应修复；
- 相比根因，改动范围过大。

### 方法三：TCP 断开时自动删除 Session

`CloseClient` 可以直接向 Worker 发送 LEAVE。

优点：

- 能快速清除断开连接对应的玩法资源。

没有采用的原因：

- 破坏现有 RECONNECT 语义；
- 随机重连和短暂网络中断都会被当作永久离开；
- 无法区分正常断线重连与显式 LEAVE。

永久断开 Session 的超时回收属于独立生命周期设计，不应混入本次修复。

### 方法四：保留读取结束状态，先解析完整帧再关闭

读取过程返回三种明确状态：

```text
kOpen       连接仍可继续读取
kPeerClosed 对端已经关闭或重置连接
kError      系统调用或协议帧错误
```

遇到 EOF 或连接断开类错误时，不立即销毁 ClientConnection，而是停止继续 `recv`，解析 `read_buffer` 中已经完整接收的帧。`HandleSocketEvent` 处理完这些帧后，再根据 `kPeerClosed` 关闭连接。

这是最终采用的方法。

## 最终实现

修改文件：

- `src/rrs/network/IOThread.h`
- `src/rrs/network/IOThread.cpp`

核心行为调整为：

1. `ReadClientFrames` 返回 `ClientReadStatus`，不再用一个 `bool` 混合表示连接状态和解析错误。
2. `recv == 0` 或连接断开类错误只将状态记为 `kPeerClosed`，然后退出接收循环。
3. 继续解析 `client.read_buffer`，将所有完整帧加入 `ready_frames`。
4. 如果对端关闭时仍有不完整尾帧，记录必要的边界错误并返回 `kError`。
5. `HandleSocketEvent` 先逐个调用 `HandleBinaryFrame`，确保 LEAVE 完成 Session 和 Worker 清理，再关闭 socket。

没有修改：

- TCP 帧布局；
- LEAVE payload；
- JOIN、RECONNECT 和 INPUT 行为；
- 断线重连保留 Session 的语义；
- Worker 和 IO 线程职责；
- Room 的最终删除条件。

## 为什么采用该方法

该方案直接修复 TCP 字节流边界上的错误，并保持改动范围最小：

- 已经成功读取的完整字节不会因为后续 EOF 或 reset 被丢弃；
- 不要求客户端通过延迟碰运气；
- 不增加协议消息和状态机；
- 不把普通断线误判成显式 LEAVE，因此保持 RECONNECT；
- 完整帧、残缺帧和系统调用错误仍有清晰区别；
- Room、SessionRegistry 和 WorkerRoomRegistry 可以继续使用现有清理链路。

与增加客户端兼容路径相比，这种修改使服务器自身正确处理 TCP 的真实行为，后续任何符合协议的客户端都能获得相同结果。

## 后续复测要求

修复前的 30 分钟日志和火焰图只能用于说明资源残留问题，不能作为最终性能基准。重新进行 1024 连接跨局测试时，应重点确认：

- 每局稳定阶段静态实体约为 65536，不能变为 131072、196608；
- 每次换局时旧房间能够归零或快速下降，新房间随后重新建立；
- 动态实体不会跨局累积；
- 客户端结束后连接数、静态实体和动态实体最终都为 0；
- CPU 和 Tick P99 不再逐局恶化；
- `reconnect_failed=0`、`server_error_received=0`、异常关闭汇总为空。

## 尚未解决的独立问题

客户端显式发送 LEAVE 的正常退出路径已经修复。客户端如果永久异常断线且不再 RECONNECT，服务器仍会按现有语义长期保留 Session 和 Room，以等待重连。

这需要单独设计 Session 超时或比赛结束后的有界清理机制，不能通过“所有 socket 关闭都等价于 LEAVE”解决，否则会破坏现有 RECONNECT 生命周期。
