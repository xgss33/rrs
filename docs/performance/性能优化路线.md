# 性能优化路线

## 目标

在 4 核 4 GB 云服务器上，以 30 Hz Tick 稳定承载 1024 个连接、约 3000 个动态实体和 60000 个静态实体，并保证房间 Tick 响应时间 P99 不超过 33.3 ms。

当前优先采用 1 个 IO 线程和 2 个 Worker 线程。只有更多线程能够稳定改善尾延迟时，才调整最终线程配置；不以耗尽全部 CPU 为目标。

## 基本原则

- 以火焰图、原始耗时样本和硬件计数器为依据，不凭代码外观判断性能。
- 每次只验证一个主要变量，使用相同负载对比修改前后结果。
- 优先删除重复工作、分配、拷贝和竞争，再考虑更复杂的数据结构或并发方案。
- 不改变协议、Tick 语义、确定性行为和客户端可观察结果。
- 优化没有稳定收益，或者收益不足以抵消复杂度时，回退修改并记录原因。
- 达到目标并保留足够的尾延迟余量后停止，不为提高平均 CPU 利用率继续增加复杂度。

## 优化顺序

### 1. 建立可重复的基准

- 固定服务器规格、线程配置、连接数、测试时长和客户端参数。
- 记录动态实体、静态实体、网络吞吐和房间 Tick 响应时间分布。
- 至少比较 P50、P95、P99、最大值和超过 33.3 ms 的比例。
- 云服务器结果出现明显波动时重复测试，避免用单次尖刺得出结论。

### 2. 删除冗余工作

- 检查同一 Tick 内的重复遍历、重复查找和重复计算。
- 检查只为下一步消费而构造的中间状态和临时容器。
- 降低非业务指标、日志和诊断代码对热路径的干扰。
- 合并自然属于同一阶段的重复处理，但不混合原本独立的职责。

### 3. 减少分配、拷贝与间接访问

- 检查每 Tick、每房间和每 Snapshot 的动态分配。
- 为容量可预测的容器合理预留空间，必要时复用明确归属的缓冲区。
- 检查玩法状态到同步状态、编码缓冲和发送队列之间的复制。
- 减少通过哈希表、多层指针和临时 ID 反复找回同一对象。
- 不为避免少量分配引入难以维护的通用对象池或缓存。

### 4. 调整容器与数据布局

- 根据实际访问方式比较连续容器、树结构和哈希结构。
- 对稠密 ID 考虑直接索引，对小规模集合考虑连续线性遍历。
- 检查结构体大小、字段顺序、对齐填充和每条缓存行容纳的实体数量。
- 将高频访问字段与低频管理字段分离，比较 AoS 与 SoA 的访问成本。
- 优先改善线性访问和缓存局部性，不单纯追求更小的结构体。

### 5. 检查线程竞争与缓存行共享

- 区分业务计算变慢、线程被抢占和共享资源等待。
- 检查多个 Worker 写入相邻 atomic、计数器或队列索引造成的伪共享。
- 让高频可写状态尽量由单个线程拥有，减少跨线程写入。
- 优先批量化和缩短临界区，再决定是否需要 SPSC 或无锁结构。
- 只对确认存在缓存行竞争的状态使用显式对齐，避免无意义地扩大对象。

### 6. 优化同步编码与网络发送

- 尽量一次确定编码缓冲大小并直接写入最终缓冲区。
- 检查公共数据是否被重复编码，以及帧在队列间是否被重复复制。
- 观察每秒系统调用次数、每次发送帧数和平均发送大小。
- 先通过消息批处理降低队列锁和唤醒次数，再评估 `writev`、`sendmsg` 或 `sendmmsg`。
- 任何缓冲复用都必须保持异步发送期间的所有权和生命周期明确。

### 7. 优化热循环与代码生成

- 将循环不变量和模式选择移到热循环外。
- 根据真实分支命中情况调整高频路径，删除玩法层中重复且无意义的校验。
- 检查高频间接调用、浮点转换、除法、取模和重复边界计算。
- 让编译器自然内联小型热函数，不机械使用强制内联。
- 代码层优化稳定后，再独立验证 LTO、PGO 和目标架构编译选项。

## 验证方法

每项优化都按以下过程验证：

1. 记录当前基准和待解释的热点。
2. 写明修改假设以及预期改善的指标。
3. 只实施能够验证该假设的最小修改。
4. 使用相同配置进行修改前后 A/B 测试。
5. 同时检查平均开销、尾延迟、错误行为和资源占用。
6. 根据结果保留、回退或记录为后续方向。

单次实验至少记录：

```text
日期与版本：
服务器配置：
IO/Worker 配置：
连接数与测试时长：
实体数量：
优化假设：
修改内容：
修改前结果：
修改后结果：
最终决定：
```

## 主要观测手段

- CPU 火焰图：确定消耗 CPU 的主要调用链。
- 房间 Tick 原始样本：计算响应时间分布和 deadline miss。
- `perf stat`：观察 IPC、cache miss、分支失败、上下文切换和 CPU migration。
- 每线程 CPU 与调度数据：区分负载分配不均、共享等待和系统抢占。
- 网络指标：观察吞吐、系统调用次数和批量发送效果。

CPU 火焰图无法直接说明锁等待和线程离核运行时间。怀疑调度或竞争时，应结合调度数据和分阶段耗时，而不是只依据函数在火焰图中的宽度。

房间 Tick 原始样本统一使用以下脚本统计，避免不同实验采用不同的百分位算法：

```bash
python3 scripts/analysis/analyze_room_tick_latency.py \
    logs/server.log \
    --connections 1024 \
    --min-dynamic-entities 2800 \
    --max-dynamic-entities 3000
```

脚本默认输出便于人工阅读的表格；大模型或其他工具使用 `--format json` 获取稳定的结构化结果。未指定负载筛选条件时会包含启动和退出阶段，因此正式对比必须明确连接数和实体数量范围。

## 停止条件

满足以下条件后结束本轮性能优化：

- 目标负载下房间 Tick 响应时间 P99 稳定低于 33.3 ms。
- deadline miss、异常关闭和服务端错误处于可接受范围。
- 线程配置在多次测试中表现稳定，并为 IO、内核和短时突发保留余量。
- 剩余热点需要明显增加复杂度，但不能提供可靠且必要的收益。

未实施但有价值的方向应记录下来，不为了完成清单而全部落地。
