# V3 - Minimal Agar-Style Game And Binary Protocol

## Goal

V3 turns the V2 server framework into a minimal playable authoritative server game.

The game is intentionally simple. The project focus remains server architecture, ownership, tick, room state, IO, and protocol boundaries.

V3 covers:

- Minimal agar-style room gameplay.
- Room capacity and room allocation policy.
- Simple custom binary protocol.

V3 does not aim to reproduce a full commercial game. It only needs enough real gameplay to validate the server pipeline.

## Game Direction

The game is a simple ball growth game:

```text
player moves
player eats food
player grows
larger player moves slower
larger player can eat smaller player
dead player respawns
largest player after 10 minutes wins
```

The server is authoritative:

- Client sends movement direction only.
- Server owns player position, speed, radius, collision, death, respawn, food spawning, and victory rules.
- Client-side prediction and interpolation are out of scope for V3.

## Core Game State

Each player entity should conceptually contain:

- `PlayerId`
- center position
- move direction
- radius
- alive/dead state
- score or mass
- respawn timing

Each food entity should conceptually contain:

- `FoodId`
- center position
- radius or value

Each room should conceptually contain:

- room id
- game state
- player entities
- food entities
- random spawn state
- game start time
- game duration
- current winner candidate

## Gameplay Rules

### Player Movement

Players move according to the latest input direction received before the current tick.

Movement speed is derived from radius:

```text
larger radius -> lower speed
```

V3 only needs a simple monotonic speed formula. Exact tuning is not important yet, but speed must have:

- base speed
- minimum speed
- radius-based slowdown

### Food

Room creates initial food when it is created or activated.

Food behavior:

- Food is placed randomly inside the room bounds.
- Player eats food when player circle overlaps food circle.
- Eating food increases player radius or mass.
- Each eaten food immediately causes one new food to spawn.

V3 does not need food movement or complex food types.

### Player Eating Player

A player can eat another player when:

- circles overlap
- attacker is meaningfully larger than victim

V3 should use a threshold to avoid equal-size ambiguity:

```text
attacker.radius > victim.radius * eat_ratio
```

Initial `eat_ratio` can be around `1.1`.

When a player is eaten:

- attacker gains mass or radius
- victim enters dead state
- victim respawns later with initial radius

### Respawn

V3 uses infinite respawn.

Respawn rules:

- Dead player does not move or collide.
- Dead player respawns after a short delay.
- Respawn position is random.
- Respawn radius resets to initial player radius.

### Match End

Each room match lasts 10 minutes.

At match end:

- winner is the largest alive player, or highest score if score is introduced
- server sends game-over result
- room can reset for a new match

V3 only needs one clear winner rule. Advanced ranking and rewards are out of scope.

## Room Capacity And Scaling

Each room supports up to 16 players.

Project target is 5000+ concurrent players:

```text
5000 players / 16 players per room ~= 313 active rooms
```

V3 should use a fixed room pool with dynamic activation:

```text
server startup
  -> create configured room count
  -> assign rooms to workers
  -> idle room waits for players
  -> active room ticks at normal rate
  -> full room is skipped by matchmaking
```

This keeps ownership simple:

- Room is still owned by one WorkerThread.
- Room does not migrate between workers in V3.
- Room creation/destruction during runtime is not required in V3.

Dynamic runtime room creation is deferred. It needs additional design for:

- RoomId allocation
- worker load balancing
- room lifecycle state
- room destruction
- room migration or rebalancing

## Matchmaking Direction

V2 uses explicit room join:

```text
JOIN <room_id> <player_id>
```

V3 should move toward server-side room assignment:

```text
MATCH
```

Expected behavior:

- IOThread receives match request.
- Server selects an active room with free capacity, or an idle room.
- Session is bound to selected room.
- Worker receives bind-player message.
- Client receives match success response.

V3 may temporarily keep explicit room join during V3.1 gameplay validation, but the final V3 direction should be matchmaking-based.

## Protocol Direction

V3 replaces the temporary text protocol with a simple custom binary protocol.

Protocol frame:

```text
uint32 length
uint8  message_type
bytes  payload
```

Length rule:

```text
length = message_type + payload byte count
```

The 4-byte length field is not included in `length`.

Byte order should be fixed for all integer fields. V3 should choose one convention and use it consistently.

Recommended convention:

```text
network byte order for integer fields
little-endian or fixed IEEE float representation only if explicitly documented
```

Because protocol is not the main project focus, V3 should avoid complicated encoding. A small manual binary codec is enough.

## Initial Message Set

Client to server:

- `C2S_MATCH`
- `C2S_INPUT`
- `C2S_RESUME`
- `C2S_QUIT`

Server to client:

- `S2C_MATCH_OK`
- `S2C_SNAPSHOT`
- `S2C_ERROR`
- `S2C_GAME_OVER`

`S2C_INPUT_ACK` is optional. V3 can skip it if snapshot frequency is enough for validation.

## Initial Payload Direction

### C2S_MATCH

Minimal payload:

```text
player_id
```

If account/login is introduced later, match should use authenticated identity instead of client-supplied `player_id`.

### C2S_INPUT

Minimal payload:

```text
direction_x
direction_y
```

V3 can use compact integer direction:

```text
int16 direction_x
int16 direction_y
```

The server should normalize or clamp the direction.

### C2S_RESUME

Minimal payload:

```text
session_id
```

Generation is still server-owned. Client should not decide generation.

### S2C_SNAPSHOT

Initial full snapshot can include:

```text
room_id
tick_seq
players
foods
```

Player snapshot:

```text
player_id
x
y
radius
alive
```

Food snapshot:

```text
food_id
x
y
radius
```

V3 should use full snapshot first. Delta snapshot, ack, and recovery are not part of V3.

## V3 Phasing

### V3.1 - Real Room Gameplay With Current Protocol

Goal:

- Replace demo movement with real agar-style room rules.
- Keep current text protocol temporarily to validate gameplay quickly.

Includes:

- player radius
- radius-based speed
- room bounds
- food spawn
- player eats food
- player eats player
- respawn
- 10-minute room match state

Does not include:

- binary protocol
- advanced matchmaking
- dynamic room creation

### V3.2 - Room Capacity And Matchmaking

Goal:

- Make room capacity and room assignment explicit.

Includes:

- 16 players per room
- fixed room pool
- active/idle/full room state
- simple room selection for match
- Worker-owned room remains unchanged

Does not include:

- room migration
- runtime room expansion
- load-aware rebalancing

### V3.3 - Simple Binary Protocol

Goal:

- Replace temporary text protocol with minimal binary framing.

Includes:

- 4-byte length prefix
- 1-byte message type
- manual payload encode/decode
- binary input
- binary match/resume
- binary full snapshot

Does not include:

- protobuf
- compression
- encryption
- delta snapshot
- reliability layer above TCP

## Important Boundaries

IOThread responsibilities:

- owns client fd
- reads protocol frames
- decodes client messages
- binds fd to session
- routes messages to worker queue
- drains worker-to-IO outbound queue
- encodes server messages
- writes to client fd

WorkerThread responsibilities:

- owns rooms
- owns room tick
- owns gameplay state
- consumes input messages
- produces snapshots
- does not access client fd
- does not wake IOThread directly

SessionRegistry responsibilities:

- owns authoritative session records
- owns session generation
- owns session-to-IO owner mapping
- protects reconnect/disconnect races

Room responsibilities:

- owns gameplay entities
- owns food
- owns collision and rule resolution
- owns match timer and winner calculation

## Deferred

The following are intentionally not part of V3:

- spatial partitioning
- interest management
- delta snapshot
- client prediction
- replay
- persistence of match result
- auth/login
- multi-process scaling
- dynamic room migration
- protobuf replacement
- UDP transport

## Acceptance Direction

V3 should be considered successful when:

- Multiple clients can match into a room.
- Room supports up to 16 players.
- Players can move, eat food, grow, slow down, eat smaller players, die, and respawn.
- Food count remains stable by respawning eaten food.
- Room can produce full snapshots containing players and food.
- Server can run continuously without fake internal clients.
- Worker does not access IO-owned state.
- IO does not access Room state directly.
- The final V3 protocol is binary-framed instead of text-based.
