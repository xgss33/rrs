# V1.5 Message Channel Refactor

## Goal

V1.5 decouples `IOThread` and `WorkerThread` by moving cross-thread queues into an explicit message channel object.

Before this iteration:

```text
IOThread -> WorkerThread::postMessage
WorkerThread -> IOThread::postOutbound through callback
```

After this iteration:

```text
IOThread -> MessageChannels::inboundToWorker -> WorkerThread
WorkerThread -> MessageChannels::outboundToIo -> IOThread
```

## Completed

- Added `MessageChannels`.
- Moved IO-to-Worker inbound queue out of `WorkerThread`.
- Moved Worker-to-IO outbound queue out of `IOThread`.
- Removed `IOThread`'s direct dependency on `WorkerThread`.
- Removed `WorkerThread`'s outbound callback dependency on `IOThread`.
- `main` now owns channel assembly and injects the same channels into both runtime components.

## Current Ownership

- `IOThread` owns sockets, client state, session state, protocol parsing, and per-client outbound queues.
- `WorkerThread` owns rooms, player entities, tick execution, input validation, and snapshot production.
- `MessageChannels` owns the cross-thread message queues.
- `main` owns component wiring.

## Message Flow

```text
Client INPUT/JOIN/RECONNECT
  -> IOThread parses command
  -> channels.inboundToWorker.push(InboundMessage)
  -> WorkerThread drains inboundToWorker
  -> WorkerThread updates room/tick state
  -> WorkerThread pushes OutboundMessage into outboundToIo
  -> IOThread drains outboundToIo
  -> IOThread sends snapshot to client
```

## Design Notes

- The queues are still mutex-based `ThreadSafeQueue`; V1.5 changes ownership boundaries, not queue implementation.
- `SessionManager` stays inside `IOThread` because session state is currently tied to socket/client protocol ownership.
- `Room` and `PlayerEntity` stay inside `WorkerThread`; IO still never mutates room state directly.
- This prepares the project for a later router/sharding layer where inbound messages can be routed to one of many workers.
