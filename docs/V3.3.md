# V3.3 - Simple Binary Protocol

## Goal

V3.3 adds the first custom binary protocol and removes the temporary text protocol path.

The binary protocol is intentionally small. Its purpose is framed message transport, not complex serialization.

## Frame Format

Each binary message uses:

```text
uint32 length
uint8  message_type
bytes  payload
```

Length rule:

```text
length = message_type byte + payload byte count
```

The 4-byte length field is not included in `length`.

Integer fields use network byte order.

Float fields in snapshot payload are encoded as IEEE-754 `float32` bit pattern in network byte order.

## Client Message Types

```text
1 = C2S_MATCH
2 = C2S_INPUT
3 = C2S_RESUME
4 = C2S_QUIT
```

### C2S_MATCH

Payload:

```text
uint64 player_id
```

### C2S_INPUT

Payload:

```text
int16 move_x
int16 move_y
```

### C2S_RESUME

Payload:

```text
uint64 session_id
```

### C2S_QUIT

Payload:

```text
empty
```

## Server Message Types

```text
101 = S2C_MATCH_OK
102 = S2C_INPUT_ACK
103 = S2C_SNAPSHOT
104 = S2C_ERROR
105 = S2C_RESUME_OK
106 = S2C_QUIT_OK
```

### S2C_MATCH_OK

Payload:

```text
uint64 room_id
uint64 session_id
uint64 generation
```

### S2C_INPUT_ACK

Payload:

```text
uint64 generation
```

### S2C_RESUME_OK

Payload:

```text
uint64 session_id
uint64 generation
```

### S2C_ERROR

Payload:

```text
bytes error_text
```

`S2C_ERROR` payload is still a short ASCII error code. This is acceptable for V3.3 because the protocol focus is framing and game data transport.

### S2C_SNAPSHOT

Payload:

```text
uint64 room_id
uint64 tick_seq
uint16 player_count
repeated players:
  uint64 player_id
  float32 x
  float32 y
  float32 radius
  uint8 alive
uint16 food_count
repeated foods:
  uint64 food_id
  float32 x
  float32 y
  float32 radius
uint8 match_over
uint64 winner_player_id
```

V3.3 still sends full snapshots only.

## Protocol Handling

There is no explicit handshake in V3.3.

All client traffic must be binary-framed.

IOThread decodes only binary frames. Text commands are not supported.

## Connection Greeting Change

V3.3 removes the automatic `OK CONNECT` greeting.

Reason:

- A binary client should not receive unexpected text before its first binary response.
- Client should send the first command immediately after TCP connect.

## Runtime Boundary

Worker still does not access IO state.

Worker produces binary snapshot payloads only.

IOThread sends binary frames only.

## Intentional Limits

- No compression.
- No protobuf.
- No encryption.
- No packet batching.
- No delta snapshot.
- No client ack or recovery.
- No UDP.

## Validation Direction

V3.3 is valid when:

- Server compiles without warnings.
- Binary client can `MATCH`.
- Binary client can send `INPUT`.
- Binary client receives binary `S2C_SNAPSHOT`.
- Binary client can `RESUME`.
- Text commands are not accepted.
