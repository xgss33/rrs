# V2.4 - Room Router And Session-Based Worker Output Routing

## Goal

V2.4 makes routing boundaries explicit.

Before this iteration:

- room-to-worker routing used the same formula in multiple places
- worker-to-IO output routing depended on `InboundMessage.io_thread_id`

That made the routing rule work, but the ownership boundary was still too implicit. Worker output should be routed by the authoritative session state, not by a copied IO id carried inside old inbound messages.

## Completed

- Added `RoomRouter`.
- Centralized the static V2 room routing rule:

```text
WorkerId = (room_id - 1) % worker_count
```

- `main` now uses `RoomRouter::OwnsRoom()` to assign rooms to workers.
- `IOThread` now uses `RoomRouter` for JOIN / INPUT / the then-current RECONNECT worker routing.
- Current runtime command set uses `RESUME <session_id>` instead of `RECONNECT` after V2.6.
- `WorkerThread` now receives `SessionRegistry&`.
- Worker snapshot output now resolves target IO through:

```text
SessionRegistry::FindIoThread(session_id)
```

- Removed the transitional `InboundMessage.io_thread_id` field.
- `InboundMessage` now only carries message data needed by worker ownership:
  - message type
  - session id
  - room id
  - player id
  - generation
  - input payload

## Current Input Flow

```text
Client command
  -> IOThread
  -> SessionRegistry create/find/read session
  -> RoomRouter::Route(room_id)
  -> worker_inbound[worker_id]
  -> WorkerThread
  -> Room
```

## Current Snapshot Output Flow

```text
Room tick
  -> WorkerThread::PublishSnapshot
  -> bound session list
  -> SessionRegistry::FindIoThread(session_id)
  -> worker_to_io[worker_id][io_thread_id]
  -> IOThread drains queues addressed to itself
  -> client outbound queue
```

## Ownership Boundary After V2.4

### Room Ownership

- `RoomRouter` defines room ownership.
- `WorkerThread` mutates only rooms assigned to itself.
- `IOThread` never mutates room state.

### Session Ownership

- `SessionRegistry` is the authoritative session state.
- `SessionRegistry` owns `SessionId -> IoThreadId`.
- Worker output routing uses the registry instead of stale copied IO metadata.

### IO Ownership

- `IOThread` owns client fds and client outbound queues.
- `IOThread` owns local fd/session cache only as a hot-path copy.

## Intentional Transitional Limits

- `Acceptor` still assigns all accepted fds to the first IOThread.
- IOThread still polls client fds instead of epoll.
- Acceptor-to-IO handoff still does not use eventfd.
- Real `RESUME <session_id>` on a new TCP connection is not implemented yet.
- Disconnected sessions can still remain bound in worker-side `bound_sessions_`; if no client owns the fd, IOThread simply cannot find the client and drops the outbound message.

The next V2 iteration should move toward Acceptor-to-IO handoff improvements.

## Validation

- Compilation only.
- No end-to-end client test is required for this intermediate V2 step.
