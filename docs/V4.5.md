# V4.5 Minimal Server Metrics

## Goal

Add a minimal server-side metrics path for pressure testing without putting slow work into IO or worker hot paths.

## Metrics

- `rrs_net_connections_current`: current TCP connections accepted by IO threads.
- `rrs_net_bytes_in_per_sec`: bytes read from sockets per second, calculated by reporter.
- `rrs_net_bytes_out_per_sec`: bytes written to sockets per second, calculated by reporter.
- `rrs_worker_tick_cost_us_last`: latest `TickDueRooms()` cost per worker.
- `rrs_worker_tick_cost_us_max_5s`: max `TickDueRooms()` cost per worker in the latest 5 second window.
- `rrs_process_cpu_percent`: process CPU usage percent. Multi-core usage can exceed 100%.
- `rrs_process_memory_rss_bytes`: process resident memory size.

## Design

`MetricsRegistry` is the only object touched by hot-path code. IO and worker threads only update atomics.

`MetricsReporter` owns an independent thread. It wakes every 5 seconds, collects a snapshot, reads `/proc/self/stat` and `/proc/self/statm`, computes derived values, and writes one structured log line.

The reporter output remains log-based for now, but metric names are stable and can be reused by a future Prometheus, HTTP, or JSON reporter.

Metrics are written through `Logger::Metrics()` and stay visible when normal startup log level is raised to `warn`, `error`, or `off`.

## Hot Path Rules

- IO and worker threads must not read `/proc`.
- IO and worker threads must not format metrics logs.
- IO and worker threads must not acquire metrics locks.
- Metrics updates in hot paths must remain atomic-only.
