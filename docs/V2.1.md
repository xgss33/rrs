# V2.1 Runtime Topology

## Goal

V2.1 starts the V2 multi-thread topology migration.

This iteration introduces multiple runtime thread instances without completing the full V2 queue, eventfd, epoll, or session registry work.

## Completed

- Added `WorkerId` and `IoThreadId` strong id types.
- Fixed the runtime topology target to 2 IOThreads.
- Kept `workerThreadCount` as the configurable worker count.
- `main` now creates multiple `WorkerThread` instances.
- `main` now creates two `IOThread` instances.
- Rooms are statically distributed across workers by:

```text
(roomId - 1) % workerThreadCount
```

- Metrics and shutdown logging now aggregate across all worker and IO thread instances.

## Intentional Transitional Limits

- `Acceptor` still hands accepted fds to the first IOThread.
- `MessageChannels` is still the V1.5 single-channel structure.
- Multiple workers currently share that transitional channel.
- Worker/IO channel matrix is not implemented yet.
- Global `SessionRegistry` is not implemented yet.
- eventfd and epoll are not implemented yet.

These are expected to be addressed by later V2 iterations.

## Validation

V2.1 only requires compilation.

No client end-to-end test is required for this step.

## Next

V2.2 should replace the single `MessageChannels` object with the planned multi-worker/multi-IO channel topology:

```text
workerInbound[workerId]
workerToIo[workerId][ioThreadId]
```
