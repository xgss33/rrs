# V3.1 - Minimal Authoritative Room Gameplay

## Goal

V3.1 replaces the previous demo room movement with a minimal authoritative agar-style game loop.

This iteration keeps the V2 text protocol so gameplay can be validated before introducing the V3 binary protocol.

## Completed

- Added real player gameplay state:
  - position
  - velocity
  - movement direction
  - radius
  - alive/dead state
  - respawn tick

- Added food entities:
  - food id
  - position
  - radius

- Room now initializes food when it is constructed.
- Room keeps food count stable by spawning one new food after one food is eaten.
- Player input now updates movement direction instead of directly teleporting position.
- Player movement is tick-based and uses `tick_interval`.
- Player speed decreases as radius grows.
- Player movement is clamped to room bounds.
- Player can eat food by circle overlap.
- Player can eat another player when:

```text
circles overlap
attacker.radius > victim.radius * eat_ratio
```

- Eaten player enters dead state.
- Dead player respawns after a short delay with initial radius and random position.
- Room tracks a 10-minute match duration and winner candidate.
- Snapshot now includes:
  - players
  - player radius
  - player alive state
  - foods
  - match-over flag
  - winner player id

## Current Protocol

V3.1 intentionally keeps the current text commands:

```text
JOIN <room_id> <player_id>
INPUT <move_x> <move_y>
RESUME <session_id>
QUIT
```

The final V3 binary protocol is deferred to V3.3.

Current text snapshot payload is expanded for validation:

```text
SNAPSHOT room=<id> tick=<tick> players=<count> foods=<count> match_over=<0|1> winner=<player_id>
```

## Runtime Ownership

The V2 ownership model remains unchanged:

- IOThread owns client fd and protocol IO.
- WorkerThread owns Room tick scheduling.
- Room owns player entities, food entities, collision, growth, death, respawn, and match state.
- Worker does not access client fd.
- IOThread does not access Room state directly.

## Intentional Limits

- No binary protocol yet.
- No matchmaking yet.
- No room capacity enforcement yet.
- No spatial partitioning.
- No client prediction.
- No delta snapshot.
- No complex ball splitting or special items.
- No persisted match result.

## Validation Direction

V3.1 is valid when:

- Server compiles.
- Client can still `JOIN`.
- Client can send `INPUT`.
- Client receives snapshots with `players` and `foods`.
- Runtime logs show room food count and player count.
- Server shuts down cleanly.
