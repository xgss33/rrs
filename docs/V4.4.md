# V4.4 Gameplay Merge

## 目标

将 `tmp/gameplay` 中确定后的最小球球大作战 gameplay 合入主干。

本版本以 Room 为 Worker 内部的玩法状态封装，Room 不访问 IO 队列，不独立拥有线程，只负责在 tick 边界消费命令、推进玩法、生成权威 snapshot。

## 最终 Gameplay

- 每个 room 最多 16 个玩家。
- 每个 room 固定 512 个食物。
- 玩家字段保持最小集合：`player_id`、`position`、`input_direction`、`radius`、`respawn_tick`。
- `respawn_tick == 0` 表示玩家存活。
- 玩家死亡后 5 秒复活，死亡期间不接受 input。
- 玩家输入只表示方向，服务端每 tick 根据方向、半径和速度公式推进位置。
- 速度公式为 `base_speed * pow(speed_reference_radius / radius, 0.4)`，结果 clamp 到 `[60, 220]`。
- 食物碰撞使用圆相交，吃掉后立即生成一个新食物。
- 玩家吞噬使用半径门槛和中心距离门槛。
- 一局 10 分钟，结束时最大半径玩家为 winner。

## Tick 顺序

Room 每 tick 的顺序固定为：

1. 消费 `entered_at < tick_start` 的命令。
2. 处理 JOIN、RECONNECT、INPUT、LEAVE。
3. 移动存活玩家。
4. 处理玩家吃食物。
5. 处理玩家吃玩家。
6. 处理复活到期玩家。
7. 更新比赛结束状态。
8. 生成一个权威 snapshot。

## 输出规则

- Worker 每个 room tick 只编码一次 snapshot。
- JOIN 成功返回 `JOIN_OK(session, generation, snapshot)`。
- RECONNECT 成功返回 `RECONNECT_OK(session, generation, snapshot)`。
- JOIN / RECONNECT 本 tick 不再额外收到普通 `SNAPSHOT`。
- LEAVE 不回包，并且不会收到本 tick 普通 `SNAPSHOT`。
- 其他 active session 收到普通 `SNAPSHOT`。

## 清理

- 删除 `SnapshotRingBuffer`，主干不再保留 snapshot history。
- 删除空的 `PlayerEntity.cpp`。
- 删除不再使用的旧 gameplay 字段和相关编码。
- 删除当前无调用方的 `Room::HasPlayer()` 和 `CirclesOverlap()`。
