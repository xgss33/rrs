# V2.5 - Acceptor To IO eventfd Handoff

## Goal

V2.5 removes the V2.4 transitional limitation where `Acceptor` always handed every accepted fd to the first IOThread.

This iteration introduces the Acceptor-to-IO handoff boundary required before epoll:

```text
accept fd
  -> select IOThread round-robin
  -> push fd into that IOThread accepted-fd queue
  -> wake that IOThread through eventfd
```

## Completed

- `Acceptor` now receives a list of IOThread targets.
- `Acceptor` selects target IOThread by round-robin.
- `Acceptor` no longer hardcodes `io_threads.front()`.
- `IOThread` now exposes accepted-fd queue and eventfd access:

```text
EnqueueAcceptedClient(fd)
wake_event_fd()
```

- Each IOThread now owns:
  - accepted fd queue
  - eventfd wake handle
  - wake/drain lifecycle

- `IOThread::Start()` creates its eventfd.
- `IOThread::Stop()` wakes the loop and closes eventfd after the thread exits.
- `EnqueueAcceptedClient()` pushes fd to the accepted queue.
- `Acceptor` writes the selected IOThread eventfd after pushing the accepted fd.
- IOThread drains eventfd and then drains accepted fd queue.
- IOThread uses `poll(eventfd, 1ms)` as a temporary wake mechanism.

## Current Handoff Flow

```text
Acceptor
  -> accept()
  -> SelectIoThread()
  -> IOThread::EnqueueAcceptedClient(fd)
  -> ThreadSafeQueue<int>::Push(fd)
  -> Acceptor writes IOThread::wake_event_fd()
  -> IOThread wakes
  -> DrainWakeEvent()
  -> AcceptNewClients()
```

## Why `poll(eventfd)` Exists In V2.5

V2.5 is not the epoll iteration yet.

However, simply writing eventfd while IOThread is sleeping would not be a real wakeup boundary. Therefore V2.5 uses a minimal `poll()` wait on eventfd so accepted fd handoff can wake the target IOThread immediately.

Full epoll ownership is still left to V2.6:

```text
epoll fd owns:
  - eventfd
  - client socket fds
```

## Intentional Transitional Limits

- IOThread still does not use epoll in V2.5. This is replaced by epoll in V2.6.
- Client sockets are still polled through the existing client map in V2.5.
- Worker-to-IO outbound messages do not yet write eventfd in V2.5.
- Real `RESUME <session_id>` is not implemented yet in V2.5. It is implemented by V2.6.
- `RECONNECT` still exists as the old same-connection command in V2.5, without fake stale input injection. Runtime code removes it in V2.6.

## Validation

- Compilation only.
- No end-to-end client test is required for this intermediate V2 step.

## Superseded By V2.6

The temporary `poll(eventfd, 1ms)` wake mechanism described in this V2.5 document is historical.

Current runtime code uses V2.6 epoll ownership:

```text
IOThread epoll fd
  -> eventfd for cross-thread wakeup
  -> client socket fds for network read/write readiness
```
