# V3.2 - Room Capacity And Simple Matchmaking

## Goal

V3.2 makes room capacity and room assignment explicit.

This iteration still keeps the temporary text protocol. Binary protocol remains deferred to V3.3.

## Completed

- Added configurable room capacity.
- Default room capacity is 16 players.
- Added a fixed room directory for matchmaking and capacity tracking.
- Server registers all configured rooms at startup.
- IOThread no longer chooses matchmaking room by touching Room state.
- RoomDirectory owns only lightweight allocation state:
  - room id
  - assigned player count
  - room capacity
  - player-to-room assignment

## Current Room Pool Model

V3.2 uses a fixed room pool:

```text
server startup
  -> create RoomDirectory
  -> register configured RoomIds
  -> create Worker-owned Room instances
  -> IOThread uses RoomDirectory for MATCH/JOIN allocation
```

Room ownership remains unchanged:

- WorkerThread owns actual Room objects.
- Room owns gameplay entities and tick state.
- IOThread owns protocol and sessions.
- RoomDirectory owns allocation metadata only.

## Capacity Rule

Each room has a fixed capacity:

```text
room_capacity = 16
```

RoomDirectory refuses new assignments when the target room is full.

Disconnected players are not released from RoomDirectory in V3.2. This preserves reconnect semantics and avoids letting a resumed player exceed capacity after another player takes the slot.

Slot release should be revisited together with match reset / room lifecycle.

## Match Command

V3.2 adds:

```text
MATCH <player_id>
```

Expected success response:

```text
OK MATCH room=<room_id> session=<session_id> generation=<generation>
```

Failure response when all rooms are full:

```text
ERR NO_ROOM_AVAILABLE
```

## Explicit Join

The old command remains available for validation:

```text
JOIN <room_id> <player_id>
```

It now goes through RoomDirectory capacity checks.

Failure response when the requested room does not exist, is full, or the player is already assigned to another room:

```text
ERR ROOM_UNAVAILABLE
```

## Respawn Timing Change

Death respawn is now fixed at 5 seconds.

The implementation derives respawn ticks from the room tick interval:

```text
respawn_delay_ticks = 5 seconds / tick_interval
```

This keeps the rule stable if server tick rate changes.

## Intentional Limits

- No dynamic room creation.
- No room destruction.
- No room migration between workers.
- No room reset slot release yet.
- No load-aware matchmaking.
- No binary protocol yet.

## Validation Direction

V3.2 is valid when:

- Server compiles.
- `MATCH <player_id>` assigns a room.
- `JOIN <room_id> <player_id>` still works but respects capacity.
- `INPUT` and snapshot still work after `MATCH`.
- `RESUME <session_id>` still works.
- Worker does not wake IOThread directly.
- IOThread does not access Room state directly for matchmaking.
