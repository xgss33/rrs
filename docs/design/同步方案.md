# Snapshot 同步方案

当前同步按实体特性分为两部分：玩家球使用每观察者 AOI 增量，食物使用房间级完整基线和位置变化广播。两部分仍编码在同一个 Snapshot 消息中。

## 玩家球同步

- `RoomVisibility` 只计算玩家球可见集合。
- 多球观察者的可见结果为所有存活球视野的并集。
- AOI 使用进入和离开距离滞回，避免边界反复增删。
- `SnapshotDeltaTracker` 为每个观察者保存上一份量化玩家状态，只发送可见掩码或量化球状态发生变化的玩家。
- 玩家离开 AOI 时通过 `removed_player_ids` 删除。
- JOIN 和 RECONNECT 使用 full reset，发送当前完整玩家可见状态。

## 食物同步

- 房间创建时固定生成 `kFoodCount` 个食物，存储顺序在房间生命周期内不变。
- `FoodIndex` 是 `foods_` 的零基下标，同时作为网格记录下标、协议身份和客户端数组下标。
- 食物不会删除；被吞噬后，同一个 `FoodIndex` 只更新位置。
- JOIN 和 RECONNECT 的 full reset 发送房间全部食物。
- 普通 Snapshot 向房间所有观察者发送本 Tick 所有位置变化的食物。
- 食物不参与 AOI，不维护每观察者可见集合，也没有食物移除列表。
- `FoodSpatialIndex` 只服务服务端食物吞噬候选查询。

## Snapshot Payload

所有整数使用大端字节序，位置和半径沿用当前量化方式。

```text
tick_seq                 uint16
flags                    uint8

player_update_count      uint8
重复 player_update_count 次：
    player_id            uint64
    visible_ball_mask    uint16
    changed_ball_mask    uint16
    按槽位顺序为 changed_ball_mask 中的每个位发送：
        position_x       int16
        position_y       int16
        radius           uint16

removed_player_count     uint8
重复 removed_player_count 次：
    player_id            uint64

food_update_count        uint16
重复 food_update_count 次：
    food_index           uint16
    position_x           int16
    position_y           int16

flags 指示存在 winner 时：
    winner_player_id     uint64
```

`flags` 当前定义：

- bit 0：full reset。
- bit 1：包含 winner。

JOIN_OK 和 RECONNECT_OK 在 Snapshot 前保留16字节 Session 前缀：`session_id uint64 + generation uint64`。

## 构建与应用

Room 在食物重新生成位置时直接记录已经量化的 `FoodSnapshotUpdate`。同一 Tick 的所有普通观察者共享该变化数组；存在 full reset 观察者时，Room 额外按需构造一份完整食物基线。

客户端应用 full reset 前清空玩家和食物状态；普通 Snapshot 对玩家应用 AOI 增量，并按 `food_index` 直接覆盖食物数组中的位置。没有玩家变化但存在食物变化时仍发送 Snapshot；两部分都没有变化时不发送。
