# 玩家实体数据结构

本文档记录玩家由单球扩展为最多 16 个球后的目标数据结构和已经确定的玩法语义。Snapshot 格式、碰撞加速算法和 AOI 不在本阶段确定。

## 目标数据结构

```cpp
inline constexpr std::size_t kMaxBallsPerPlayer = 16;

struct PlayerBall {
    Vector2 position;    // 球的位置，8 字节
    float radius{0.0F};  // 球的半径，4 字节
};  // 12 字节

struct PlayerEntity {
    PlayerId player_id;       // 玩家身份，8 字节
    Vector2 input_direction;  // 当前移动方向，由该玩家的所有球共享，8 字节
    TickSeq respawn_tick{0};  // 玩家级复活时间，8 字节

    std::uint16_t active_ball_mask{0};                 // 第 i 位对应 balls[i]
    std::array<PlayerBall, kMaxBallsPerPlayer> balls{};
};  // 当前 64 位布局下为 224 字节
```

`PlayerBall` 只保留位置和半径。当前玩法不需要球级输入、速度、冲刺状态、合并时间或独立 `BallId`。球不会在槽位之间搬移，因此一个球在服务端和客户端之间可以使用 `(player_id, slot_index)` 作为稳定身份。

`active_ball_mask` 是球槽位是否有效的唯一依据：

- 第 `i` 位为 1，表示 `balls[i]` 存活；为 0，表示该槽位空闲。
- 球数量通过有效位数量获得，不额外维护 `ball_count`。
- 玩家是否存活由掩码是否为 0 判断；只有全部球消失后才进入复活流程。
- 分裂开始时保存掩码快照，可以保证本帧新生成的球不会再次分裂。
- 空闲槽位也可以由掩码直接计算，不需要给每个球增加 `alive` 字段。

### 为什么选择 AoS

当前热点逻辑通常同时需要一个球的位置和半径：移动时半径决定速度和边界，碰撞、吞噬和空间索引同样同时使用两者。`PlayerBall` 采用 AoS 后，一次连续读取即可得到这两个字段；将位置和半径拆成多个数组没有明确收益，反而增加索引和维护成本。

`PlayerBall` 为 12 字节，16 个球槽位共 192 字节，相当于 3 条 64 字节缓存行的容量。数组起始地址不一定按缓存行对齐，实际可能接触 3～4 条缓存行；当前没有必要为此增加填充或强制 16 字节对齐。

### 为什么选择固定槽位

- 每个玩家最多只有 16 个球，固定数组的空间成本可控。
- 避免分裂、吞噬过程中进行动态内存分配。
- 槽位索引稳定，便于后续 Snapshot 表达球的新增、更新和删除。
- 16 个玩家时，`PlayerEntity` 数据约为 3.5 KiB，当前内存压力可以忽略。

## 玩法取舍

如需增加本文档没有提到但实现必需的状态或规则，需要先评估。当前优先保证逻辑简单、遍历稳定和性能可分析，不为完整游戏体验增加额外机制。

明确不实现：

- 分裂冲刺过程、速度和路径碰撞。
- 分裂后的合并时间与主动合并。
- 球之间的推挤、反弹和位置分离。
- 复杂的碰撞候选排序或游戏性优先级。

发生冲突时采用确定的遍历顺序和即时状态更新，后续遍历读取更新后的状态。

## 分裂算法

一次分裂指令处理分裂开始时的所有存活球：

1. 保存 `source_mask = active_ball_mask`，后续只遍历 `source_mask` 中的球。
2. 根据 `active_ball_mask` 计算临时 `free_mask`，用于寻找空闲槽位。
3. 按槽位从小到大遍历 `source_mask`。
4. 检查当前球分裂后的半径是否满足最小半径规则。
5. 满足规则时，从 `free_mask` 中选择最低的空闲槽位，随后同时更新 `free_mask` 和 `active_ball_mask`。
6. `free_mask` 为空时，后续不会再出现空闲槽位，可以直接结束本次分裂。

使用单独的 `source_mask` 和 `free_mask` 后，不需要为每个父球重新遍历全部 16 个槽位，同时能够保证新球本帧不会递归分裂。

正常情况下球数量按 `1 → 2 → 4 → 8 → 16` 增长。如果各球半径已经不同，过小的球可以分裂失败，其他满足条件的球仍按遍历顺序完成分裂。

## 分裂规则

父球分裂前半径为 `original_radius`，父球和子球分裂后的半径相同：

```cpp
child_radius = original_radius / std::sqrt(2.0F);
```

这样可以保证分裂前后的圆面积总和不变。仅当：

```cpp
child_radius >= room_rules::kInitialPlayerRadius
```

时允许该球分裂。

位置规则：

- 父球中心位置保持不变，只修改半径。
- 当前输入方向非零时，使用归一化后的当前输入方向。
- 当前没有输入时，固定使用 `(1, 0)`，不额外保存上一次输入方向。
- 子球直接生成在简化后的冲刺结束位置，不模拟中间过程。

```cpp
spawn_distance = child_radius * room_rules::kSplitSpawnDistanceRatio;
spawn_position = parent_position + split_direction * spawn_distance;
```

`kSplitSpawnDistanceRatio` 初始取 `3.0F`。其中 `2 × child_radius` 是两个等大球相切的中心距离，额外的 `1 × child_radius` 表示省略的冲刺位移。生成后按照子球半径将位置限制在房间边界内，不检测生成路径上的碰撞。

## 吞噬规则

同一玩家和不同玩家的球共用一套球吞噬算法：

- 不根据所有者改变半径比例、中心距离等吞噬条件。
- 调用方传入增长系数；初版友方球和敌方球都使用当前的 `kPlayerGrowthRatio`。
- 吞噬条件和半径增长公式沿用当前玩家吞噬算法，若要调整需要单独评估。
- 吞噬成功后立即清除被吞噬球对应的有效位，并更新攻击球半径。
- 被吞噬玩家仍有其他有效球时继续存活；最后一个球消失后才设置玩家级复活状态。
- 未达到吞噬条件时不推开两个球，允许重叠。

食物吞噬暂时保留独立流程。食物没有玩家球槽位和玩家生命周期，现阶段不为复用公式强行合并；如果实现时确认几何判定存在自然的公共部分，再提取最小公共逻辑。

## Tick 顺序

```text
处理输入
→ 执行分裂
→ 移动所有存活球
→ 处理食物吞噬
→ 处理所有玩家球之间的吞噬
→ 处理全灭玩家的复活
→ 更新比赛状态
→ 生成 Snapshot
```

新生成的球在当前 Tick 内不能再次分裂，但会参与后续所有行为，包括移动、食物吞噬、友方与敌方球吞噬、比赛状态计算和 Snapshot 生成。
