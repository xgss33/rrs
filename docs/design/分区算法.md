• 如果要把它做成秋招亮点，亮点不能只是“用了网格”，而应该是：

  > 根据固定房间、高频动态实体、多种空间查询、确定性和 CPU Cache 需求，设计统一的、分层的、可复用空间分区，并让碰撞、AOI
  > 和 Snapshot 都建立在同一个空间事实之上。

  最底层需要认真决定三件事：

  1. 实体如何登记到 Cell。
  2. 网格如何更新。
  3. Cell 内数据如何存储。

  下面给出对比和明确推荐。

  # 一、实体如何登记到 Cell

  ## 方案 A：按中心点只登记一个 Cell

  BallHandle 只存在于球心所在 Cell
  FoodIndex 只存在于食物中心所在 Cell

  优点：

  - 一个实体只有一个 Cell 引用；
  - 更新和重建简单；
  - 没有重复候选；
  - 内存最少。

  问题：

  假设一个大球的中心在远处，但球体边缘已经进入当前查询范围：

  查询区域        大球
  ┌──────┐      ◯
  │      │    ◯   ◯
  └──────┘      ●  球心在范围外

  如果只按中心查询，就必须额外知道“所有目标实体的最大半径”，扩大查询范围：

  query_radius += global_max_ball_radius

  一旦出现大球，很多查询都会被迫扫描大量额外 Cell。

  ## 方案 B：按 AABB 登记到所有覆盖 Cell

  球的 AABB 覆盖几个 Cell，就在几个 Cell 中登记引用：

  ┌───┬───┬───┐
  │   │ ○ │   │
  ├───┼───┼───┤
  │ ○ │ ● │ ○ │
  ├───┼───┼───┤
  │   │ ○ │   │
  └───┴───┴───┘

  优点：

  - 只要两个球可能碰撞，它们一定至少共享一个 Cell；
  - 不需要依赖全局最大半径；
  - 大球进入 AOI 时，即使球心还在视野外，也能被找到；
  - 食物碰撞、球碰撞和玩家球 AOI 使用同一种空间覆盖规则；
  - 上层只需对候选做精确圆形判断。

  代价：

  - 一个实体可能在多个 Cell 中出现；
  - 查询结果可能包含重复 Handle；
  - 碰撞候选对需要排序、去重；
  - 大球会增加 Cell 引用数量。

  ## 推荐

  推荐方案 B：AABB 多 Cell 覆盖。

  原因不是它更复杂，而是它能提供一个明确的正确性保证：

  > 任意两个实际相交的空间包围盒，都必然至少共享一个网格 Cell。

  这使碰撞和 AOI 不需要额外依赖“最大实体半径”等全局状态。

  # 二、网格如何更新

  ## 方案 A：增量 Insert/Remove/Move

  实体移动时：

  从旧 Cell 删除
  插入新 Cell
  更新反向索引

  优点：

  - 静止实体没有重建成本；
  - 少量实体移动时理论上便宜。

  问题：

  - AABB 跨多个 Cell 时，需要维护一个实体所属的全部 Cell；
  - 删除需要反向位置索引；
  - swap erase 会改变遍历顺序；
  - 分裂、死亡、复活会产生复杂生命周期；
  - 很容易出现残留引用或重复登记；
  - 每个 Cell 的小 vector 会频繁修改；
  - 确定性和缓存局部性较差。

  ## 方案 B：每个空间阶段整体重建

  每次从 Room 权威数据重新生成空间分区：

  遍历权威实体
  → 计算 AABB
  → 统计 Cell 引用数
  → 前缀和
  → 连续写入

  复杂度：

  O(Cell 数量 + 空间引用数量)

  优点：

  - 不需要维护旧 Cell；
  - 不存在残留索引；
  - 分裂、死亡、复活天然反映在下一次重建；
  - 输出是不可变的空间快照；
  - 连续内存，适合 CPU Cache；
  - 构建顺序可控，容易保证确定性；
  - 碰撞、AOI、Snapshot 在同一空间 Epoch 上读取。

  代价：

  - 静止食物也需要参与重建；
  - 如果一个 Tick 中存在多个空间阶段，可能需要重建两次。

  ## 当前规模估算

  每房间：

  食物：1024
  玩家球：最多 256
  总实体：约 1280

  即使每 Tick 重建两次、30Hz：

  1280 × 2 × 30 = 76800 个实体处理/房间/秒

  这个规模很小。相比网络序列化、玩家碰撞和系统调用，重建不是主要压力。

  ## 推荐

  推荐方案 B：空间阶段整体重建。

  这是很好的面试设计点：

  > 对高频动态但总量受限的实体，不盲目采用复杂增量维护，而是利用整体重建获得确定性、无陈旧索引和连续内存。

  # 三、Cell 内如何存储

  ## 方案 A：每个 Cell 一个 vector

  struct Cell {
      std::vector<Handle> entries;
  };

  优点：

  - 实现直观；
  - 调试方便。

  问题：

  - 1024 个 Cell 就有 1024 个 vector；
  - 大量 vector 为空；
  - 存在小内存分配；
  - Cell 数据分散；
  - 查询多个 Cell 时会跳转到不同内存块。

  适合作为原型，不适合作为最终亮点。

  ## 方案 B：扁平连续桶

  std::vector<std::uint32_t> cell_offsets;
  std::vector<std::uint32_t> cell_record_indices;

  例如：

  cell_offsets = [0, 2, 2, 5, 6]

  cell 0 = indices[0..2)
  cell 1 = indices[2..2)
  cell 2 = indices[2..5)
  cell 3 = indices[5..6)

  所有 Cell 引用位于一段连续内存中。

  构建过程：

  1. 统计每个 Cell 需要多少引用。
  2. 前缀和生成 cell_offsets。
  3. 第二次遍历，把引用写入对应位置。

  这本质上是一次稳定的 Counting Sort。

  优点：

  - 构建是线性复杂度；
  - 查询 Cell 得到一个 span；
  - 没有 Cell 级小分配；
  - 连续读取；
  - 容易预分配并复用内存；
  - 很适合作为简历中的数据布局设计点。

  ## 推荐

  直接采用方案 B，不做每 Cell vector 的过渡版本。

  # 四、避免多 Cell 登记导致数据复制

  AABB 多 Cell 覆盖不应该复制完整记录。

  推荐分成两层：

  template <typename Handle>
  struct SpatialRecord {
      Handle handle;
      Aabb bounds;
  };

  每个权威实体只有一个派生 Record：

  records:
  [BallA, BallB, BallC]

  Cell 中只存 Record 的索引：

  cell_record_indices:
  [0, 0, 1, 2, 2, 2]

  因此：

  - BallA 只有一个 SpatialRecord；
  - BallA 可以被多个 Cell 引用；
  - 每个 Cell 引用只需一个 uint32_t；
  - AABB 不会在多个 Cell 中重复复制；
  - Handle 可以回到 Room 权威数据。

  完整结构：

  template <typename Handle>
  class UniformGridLayer {
  public:
      void Rebuild(std::span<const SpatialRecord<Handle>> records);

      [[nodiscard]] std::span<const std::uint32_t>
      RecordIndicesInCell(CellIndex cell) const;

      [[nodiscard]] const SpatialRecord<Handle>&
      record(std::uint32_t index) const;

  private:
      std::vector<SpatialRecord<Handle>> records_;
      std::vector<std::uint32_t> cell_offsets_;
      std::vector<std::uint32_t> cell_record_indices_;
      std::vector<std::uint32_t> rebuild_offsets_;
  };

  这里存在两种“一对一”关系：

  权威实体 1:1 SpatialRecord
  SpatialRecord 1:N Cell 引用

  既保证唯一派生记录，又支持大实体覆盖多个 Cell。

  # 五、统一布局，分开类型 Layer

  ## 混合容器

  struct SpatialHandle {
      EntityType type;
      uint32_t index;
  };

  Food 和 Ball 混在同一个数组。

  优点：

  - AOI 扫描一次即可拿到全部类型。

  问题：

  - 食物碰撞需要不断判断类型；
  - 玩家碰撞也需要过滤类型；
  - 分支增加；
  - Handle 变大；
  - 类型安全差；
  - 不同实体的数据投影无法独立优化。

  ## 独立网格

  Food 和 Ball 各自维护房间尺寸、Cell 大小和坐标映射。

  问题：

  - 重复空间规则；
  - 容易出现 Cell 参数不一致；
  - AOI 要处理两套 CellRange；
  - 本质上没有真正统一。

  ## 推荐：公共 Layout + 类型 Layer

  class RoomSpatialPartition {
  private:
      UniformGridLayout layout_;
      UniformGridLayer<FoodIndex> foods_;
      UniformGridLayer<PlayerBallHandle> balls_;
  };

  两层共享：

  - 房间边界；
  - Cell 大小；
  - Cell 坐标；
  - AABB 到 CellRange 的算法；
  - Cell 遍历顺序。

  但数据分开连续存储。

  这兼顾了统一性、类型安全和 CPU Cache。

  # 六、推荐的最底层最终模型

  UniformGridLayout
      固定房间边界
      坐标 → Cell
      AABB → CellRange
      CellCoord → CellIndex

  UniformGridLayer<Handle>
      唯一 SpatialRecord 数组
      Cell Offset 数组
      连续 RecordIndex 数组
      整体稳定重建
      不可变查询

  RoomSpatialPartition
      组合公共 Layout
      Food Layer
      PlayerBall Layer

  底层不提供：

  - QueryNearbyFood
  - QueryVisiblePlayers
  - ResolveCollision
  - BuildSnapshot

  只提供：

  CellRange CellsForBounds(Aabb bounds) const;
  std::span<const uint32_t> RecordIndicesInCell(CellIndex cell) const;
  const SpatialRecord<Handle>& Record(uint32_t index) const;

  语义全部由上层决定。

  # 七、命名和文件建议

  不使用 GridManager、SpatialSystem、EntityGrid 这类职责模糊的名字。

  建议：

  include/rrs/spatial/SpatialTypes.h
  include/rrs/spatial/UniformGrid.h
  include/rrs/game/RoomSpatialPartition.h

  src/spatial/UniformGrid.cpp
  src/game/RoomSpatialPartition.cpp

  类型：

  Aabb
  GridCellCoord
  GridCellRange
  GridCellIndex
  SpatialRecord<Handle>
  UniformGridLayout
  UniformGridLayer<Handle>
  PlayerBallHandle
  RoomSpatialPartition

  UniformGrid 表达算法，RoomSpatialPartition 表达游戏层组合职责。

  # 八、为什么这能成为亮点

  这套设计可以在面试中完整讲出一条推导链：

  1. Room 保持唯一权威数据，避免空间索引成为第二份状态。
  2. 使用 AABB 多 Cell 覆盖保证候选完整性。
  3. 使用整体重建避免复杂增量维护和陈旧索引。
  4. 使用 Counting Sort/CSR 式连续桶改善 CPU Cache。
  5. 使用唯一 Record + Cell RecordIndex 避免多 Cell 数据复制。
  6. 使用公共 Layout + 类型 Layer 支撑不同实体查询。
  7. 碰撞通过稳定候选排序保持确定性。
  8. AOI 根据 CellRange 生成可见集合。
  9. Snapshot 可以按 AOI Cell/ViewKey 缓存和共享编码。
  10. 通过候选数量、重建耗时和带宽数据验证收益。

  ## 我的最终推荐

  - 固定稠密均匀网格；
  - 房间范围固定 [-1024, 1024]；
  - Cell 大小先固定 64；
  - AABB 多 Cell 覆盖；
  - 每个空间阶段整体重建；
  - Counting Sort 扁平桶；
  - 唯一 SpatialRecord；
  - Cell 只保存 RecordIndex；
  - 公共 Layout + Food/Ball 类型 Layer；
  - 查询结果允许重复，上层稳定排序去重；
  - 第一阶段不缓存额外玩法字段。

  需要你确认的就是三个核心决定：

  1. 接受 AABB 多 Cell 覆盖，而不是中心点单 Cell。
  2. 接受整体重建，而不是增量 Move/Remove。
  3. 接受公共 Layout + 类型 Layer，而不是混合实体容器。
