# V4.7 Dirty Food Snapshot

## Goal

Reduce downstream bandwidth by stopping normal tick snapshots from sending all food entities every frame.

## Food Model

- Each room creates `1024` food entities on initialization.
- Food ids are fixed for the lifetime of the room.
- When a food is eaten, the same food id is kept and only its position changes.
- Food radius is fixed and is no longer encoded in snapshots.

## Snapshot Semantics

`RoomSnapshot.foods` means "the food states included in this message".

- `JOIN_OK` and `RECONNECT_OK` carry full snapshots, so `foods` contains all room foods.
- Normal `SNAPSHOT` messages carry delta snapshots, so `foods` contains only dirty foods whose positions changed during the tick.
- `Room` decides whether a tick result needs a full snapshot. `Worker` only encodes and sends the snapshots already produced by `Room`.

Clients can process both cases with the same rule:

```text
for each food in snapshot.foods:
    local_food_position[food_id] = position
```

No `SnapshotKind` field is needed because TCP preserves order and reconnect receives a full snapshot.

## Protocol Change

Food snapshot encoding changed from:

```text
uint64 food_id
float position_x
float position_y
float radius
```

to:

```text
uint64 food_id
float position_x
float position_y
```
